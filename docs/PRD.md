# PRD - LINE LIFF 點餐系統

## 文件目的

本 PRD 定義 LINE LIFF 點餐系統第一版的可執行產品範圍。

本文件聚焦產品行為與業務規則。詳細資料庫結構、API 規格、驗收測試與 LINE 設定刻意延後，後續在使用者要求時由 Claude Code 補齊。

## 產品目標

	
客人可以從 LINE 官方帳號開啟點餐頁，瀏覽菜單、加入購物車、選擇取餐資訊、送出訂單、取得訂單編號，並於取餐時到店付款。

店家人員可以查看與管理訂單。廚房人員可以清楚看到新訂單與製作中訂單。

## 產品原則

- 先完成可靠 MVP，再擴充進階功能。
- 客人流程要短、清楚、適合手機操作。
- 後台與廚房畫面要簡單且易讀。
- 第一版只支援到店付款。
- MVP 不做完整 POS、付款、會員、優惠券、外送或多分店功能。
- 不要只因為參考截圖出現某功能，就把它做進產品。
- 參考截圖只作為流程範例；店名、分類、商品、圖片、選項與價格都必須能替換成商家自己的菜單資料。

## 使用者與工作

### 客人

客人想要：

- 從 LINE 開啟點餐頁。
- 透過分類或搜尋快速找到商品。
- 理解商品價格與圖片。
- 選擇數量與必填商品選項。
- 查看購物車小計。
- 選擇取餐方式與取餐時間。
- 輸入聯絡資訊。
- 送出訂單。
- 取得訂單編號。
- 到店付款並取餐。

### 店家 / 後台人員

店家人員想要：

- 查看今日新訂單。
- 開啟訂單明細。
- 理解商品、數量、選項、備註、取餐時間、客人姓名與電話。
- 更新訂單狀態。
- 必要時取消訂單。

### 廚房人員

廚房人員想要：

- 查看需要製作的訂單。
- 不用打開太多畫面就能讀到商品名稱、數量、必填選項與備註。
- 知道哪些訂單是新訂單，哪些已經在製作中。	

## MVP 範圍

### 客人 LIFF

MVP 客人端功能：

- 從 LINE 官方帳號開啟 LIFF 點餐 app。
- 顯示菜單分類。
- 顯示包含圖片、名稱與價格的商品卡片。
- 搜尋商品。
- 開啟商品詳情彈窗。
- 選擇商品數量。
- 選擇必填商品選項。
- 將選定商品加入購物車。
- 查看購物車數量與小計。
- 加入商品後可繼續選購。
- 進入結帳。
- 結帳步驟 1：選擇取餐方式與取餐時間。
- 結帳步驟 2：填寫客人資訊與備註。
- 送出訂單。
- 顯示包含訂單編號與取餐摘要的訂單完成頁。

### 店家 / 後台

MVP 後台功能：

- 查看今日訂單。
- 依狀態篩選或視覺分組訂單。
- 查看訂單明細。
- 更新訂單狀態。
- 取消訂單。

### 廚房顯示

MVP 廚房功能：

- 查看 `pending` 與 `preparing` 訂單。
- 顯示訂單編號、建立時間、取餐時間、商品列表、數量、商品選項與備註。
- 讓新訂單容易被注意到。
- 讓長備註可閱讀。

### 商家菜單資料

MVP 需要商家提供的菜單資料。系統不得永久綁定參考截圖中的示範店家、分類、商品、照片或價格。

菜單資料必須能表示：

- 分類名稱。
- 商品名稱。
- 商品圖片。
- 商品價格。
- 商品可售狀態。
- 必填或選填的商品選項群組。
- 選項標籤與可選的價格加價。

除非另行要求，第一版不必包含完整菜單管理 UI。MVP 可以透過 seed data、JSON/CSV 匯入、後台最小維護資料來源，或在實作規劃時經核准的其他簡單方式，載入商家自己的菜單。

實作上應將菜單資料與 UI 程式碼分離，讓不同店家可以替換菜單，而不需要重寫點餐流程。

## 明確不在 MVP 範圍內

除非使用者明確要求，MVP 不要實作：

- 線上付款。
- LINE Pay。
- 信用卡付款。
- 發票串接。
- 會員點數。
- 優惠券。
- 外送。
- 多分店管理。
- 複雜庫存管理。
- AI 推薦。
- 員工排班。
- 會計系統。
- 完整 POS 系統。
- 揪團點餐。
- 獨立客人訂單查詢入口。
- 完整菜單管理儀表板。

## 業務規則

### 店家與取餐

- MVP 支援單一店家與單一門市。
- 店家顯示名稱可以在本機示範資料中使用 `兄弟鹽酥雞-林口街店`，直到真實店家設定存在。
- 正式環境的店名、地址、電話、分類、商品、價格、圖片與選項必須是商家可配置資料。
- MVP 取餐方式只有 `自取`。
- 客人取餐時到店付款。
- 客人送出訂單前必須選擇取餐時間。
- 若有店家地址與電話，結帳時應顯示。

### 菜單瀏覽

- 商品屬於分類。
- 客人可以從分類標籤切換分類。
- 客人可以打開分類抽屜選單跳到指定分類。
- 客人可以用關鍵字搜尋全部商品。
- 不可售商品不得被下單。
- 商品卡片至少要顯示圖片、名稱與價格。

### 商品詳情與選項

- 點擊商品卡片會開啟商品詳情彈窗。
- 商品詳情必須顯示圖片、名稱、價格、數量選擇器與操作按鈕。
- 加入商品時數量不可低於 1。
- 商品加入購物車或繼續結帳前，必填選項群組必須完成選擇。
- 選項可以表示辣度、胡椒量或其他餐點需求。
- 系統應支援商品專屬選項規則，包括必填群組與選填群組。
- 選項加價可以是 0。

### 購物車

- 購物車儲存已選商品、數量與已選選項。
- 同一商品若選項不同，應視為不同購物車品項。
- 客人可在結帳前修改數量或移除商品。
- 前端顯示的購物車小計是 UX 用估算。
- 後端存在時，最終訂單總金額必須由伺服器計算。

### 結帳

結帳有兩個客人可見步驟。

步驟 1：取餐方式與取餐時間

- 顯示店名。
- 顯示商品摘要與小計。
- 讓客人選擇取餐方式。
- MVP 只支援 `自取`。
- 讓客人選擇取餐時間。
- 未完成步驟 1 必填欄位，不可進入步驟 2。

步驟 2：訂購資訊

- 顯示取餐方式、取餐時間、店家地址與店家電話。
- 顯示商品摘要與小計。
- 要求客人填寫：
  - 訂購人，必填。
  - 手機號碼，必填。
  - 備註，選填，最多 200 字。
- 未填必填客人資訊不可送出。
- 空購物車不可送出。

### 訂單建立

- 客人送出結帳時，系統建立一筆訂單。
- 系統產生唯一訂單編號。
- 訂單初始狀態為 `pending`。
- 訂單應儲存商品名稱、商品價格、已選選項、數量與備註作為快照，避免未來菜單變更影響歷史訂單。

詳細 API 與資料庫行為之後應定義在 `docs/API.md` 與 `docs/DB_SCHEMA.md`。

## 訂單狀態

| 狀態 | 意義 | 說明 |
| --- | --- | --- |
| `pending` | 新訂單 | 訂單已送出，等待店家處理。 |
| `preparing` | 製作中 | 廚房或店家已開始製作。 |
| `ready` | 可取餐 | 餐點已完成，客人可以取餐。 |
| `picked_up` | 已取餐 | 客人已付款並取餐。 |
| `cancelled` | 已取消 | 訂單已取消，但仍保留在歷史紀錄。 |

MVP 允許的狀態轉換：

- `pending` -> `preparing`
- `pending` -> `cancelled`
- `preparing` -> `ready`
- `preparing` -> `cancelled`
- `ready` -> `picked_up`
- `ready` -> `cancelled`

## 後台規則

- 後台訂單列表預設顯示今日訂單。
- 新訂單應容易被注意到。
- 後台應顯示訂單編號、狀態、建立時間、取餐時間、客人姓名、電話、總金額與主要商品摘要。
- 後台明細頁應顯示完整商品列表、數量、已選選項、商品備註、訂單備註、取餐資訊，以及可用時的狀態歷史。
- 後台狀態更新應是明確操作，避免單次模糊點擊造成誤更新。

## 廚房規則

- 廚房畫面優先顯示 `pending` 與 `preparing` 訂單。
- 廚房預設不需要看到已完成或已取消訂單。
- 廚房應明顯呈現選項與備註。
- 廚房畫面應能在工作距離下閱讀。
- 當技術架構支援時，廚房畫面應支援頻繁刷新或即時更新。

## 錯誤與空狀態

客人端 UI 應處理：

- 菜單載入中。
- 菜單載入失敗。
- 分類沒有商品。
- 搜尋沒有結果。
- 商品不可售。
- 必填選項未選。
- 空購物車。
- 結帳送出失敗。
- LIFF 初始化失敗。

後台與廚房 UI 應處理：

- 今日沒有訂單。
- 訂單載入失敗。
- 狀態更新失敗。
- 訂單已被其他使用者變更。

## 成功定義

MVP 成功條件：

- 客人可以在手機上完成完整 LIFF 點餐流程。
- 客人取得唯一訂單編號。
- 店家後台可以看到訂單並更新狀態。
- 廚房可以看到需要製作的訂單。
- 訂單送出後，總金額與商品快照保持穩定。
- 不包含線上付款。
- 核心畫面在真實店家使用情境下仍清楚易懂。
