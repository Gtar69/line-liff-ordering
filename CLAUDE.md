## 專案概覽

本專案是一個給餐飲商家使用的 LINE LIFF 點餐系統。

客人從 LINE 官方帳號開啟 LIFF app，瀏覽菜單、加入購物車、送出訂單、取得訂單編號，並於到店取餐時付款。

第一版重點是做出簡單、穩定的手機點餐流程。

參考截圖只作為示意。它們定義的是有用的流程模式，不是固定的商家內容。不要把截圖中的店名、分類、商品、圖片或價格寫成固定產品需求。

## 核心產品規則

- 這是一個 LINE LIFF 點餐系統。
- MVP 只支援單一店家與單一門市。
- MVP 只支援到店自取。
- MVP 只支援到店付款。
- 除非使用者明確要求，否則不要加入線上付款。
- 除非使用者明確要求，否則不要加入 LINE Pay。
- 除非使用者明確要求，否則不要加入信用卡付款。
- MVP 不要加入發票、會員點數、優惠券、外送、多分店管理或完整 POS 功能。
- 第一版要保持簡單且穩定。
- 優先處理訂單建立、廚房可視性與店家訂單管理。
- 客人端 UI 必須行動優先。
- 店家端與廚房畫面應簡單、快速、容易閱讀。
- 商家必須可以使用自己的菜單資料。

## MVP 範圍

### 客人端

客人應該可以：

- 從 LINE 官方帳號開啟 LIFF app。
- 依分類查看菜單品項。
- 搜尋菜單品項。
- 開啟商品詳情。
- 選擇商品數量。
- 選擇必填商品選項，例如辣度或餐點需求。
- 將商品加入購物車。
- 編輯購物車內容。
- 查看小計。
- 選擇取餐方式。
- 選擇取餐時間。
- 輸入訂購人姓名。
- 輸入手機號碼。
- 填寫最多 200 字的訂單備註。
- 確認並送出訂單。
- 送出後看到訂單編號。

### 店家 / 後台端

店家應該可以：

- 查看今日訂單。
- 查看訂單明細。
- 更新訂單狀態。
- 取消訂單。

商家必須能提供自己的菜單資料。第一版可以透過 seed data、JSON/CSV 匯入、後台最小維護資料來源，或其他簡單設定方式做到。除非使用者明確要求，否則完整菜單管理 UI 不是必須；但菜單內容不得永久寫死為參考截圖中的內容。

### 廚房端

廚房應該可以：

- 查看 `pending` 與 `preparing` 訂單。
- 清楚看到商品名稱、數量、商品選項與備註。
- 快速辨識訂單狀態。

## MVP 不包含

除非使用者明確要求，否則不要實作：

- 線上付款
- LINE Pay
- 信用卡付款
- 發票串接
- 會員點數
- 優惠券
- 外送
- 多分店管理
- 複雜庫存管理
- AI 推薦
- 員工排班
- 會計系統
- 完整 POS 系統
- 揪團點餐
- 獨立客人訂單查詢入口

如果參考截圖中出現上述功能，請只把它當成參考，不要在 MVP 中實作，除非使用者同意。

如果參考截圖中出現特定商品、分類、價格、照片或店家資訊，請只把它們當成範例。

## 訂單狀態

使用以下訂單狀態：

| 狀態 | 意義 | 可見對象 |
| --- | --- | --- |
| `pending` | 新訂單 | 後台、廚房 |
| `preparing` | 製作中 | 後台、廚房 |
| `ready` | 可取餐 | 後台 |
| `picked_up` | 已取餐 | 後台 |
| `cancelled` | 已取消 | 後台 |

除非必要且經過核准，不要自行新增其他狀態。

MVP 允許的狀態轉換：

- `pending` -> `preparing`
- `pending` -> `cancelled`
- `preparing` -> `ready`
- `preparing` -> `cancelled`
- `ready` -> `picked_up`
- `ready` -> `cancelled`

## UI 實作規則

客人端畫面以 `docs/UI_FLOW.md` 作為唯一真實來源。

客人 LIFF UI：

- 行動優先。
- 大型觸控目標。
- 清楚的商品卡片。
- 清楚的分類導覽。
- 清楚的購物車小計。
- 清楚的結帳步驟。
- 清楚顯示訂單編號的完成頁。
- 避免複雜導覽。
- 避免桌機優先版面。

後台 UI：

- 優先呈現今日訂單。
- 清楚顯示訂單狀態。
- 讓狀態更新容易操作。
- MVP 避免不必要的儀表板。

廚房 UI：

- 文字要大且容易閱讀。
- 清楚顯示訂單編號。
- 以訂單為單位分組商品。
- 明顯呈現備註與商品選項。
- 避免雜亂。

## 常用 Plugins / MCP 使用規則

Claude Code 可使用以下常用工具輔助開發。若工具在當前環境不可用，請清楚回報並改用可行替代方式；不要假裝已使用。

### Superpowers

適用情境：

- 新功能規劃。
- TDD。
- 系統化 debug。
- 安全重構。
- 完成前自我審查。

使用規則：

### frontend-design

適用情境：

- 客人端 LIFF 頁面。
- 菜單瀏覽、商品詳情、購物車、結帳流程。

檢查重點：

- 手機版面是否行動優先。
- 按鈕與觸控目標是否足夠大。
- 商品卡片是否清楚。

- 錯誤、載入、空狀態是否完整。
- 廚房畫面是否足夠易讀。

### chrome-devtools MCP

適用情境：

- 前端本機除錯。
- LIFF 頁面瀏覽器測試。
- Console 錯誤、network request、版面問題。
- 手機 viewport 驗證。

檢查重點：

- Console 是否有錯誤。
- API request / response 是否符合預期。
- 行動版是否有遮擋、溢出或不可點擊問題。
- 表單驗證、載入狀態與錯誤狀態是否正確。

### pr-review-toolkit

適用情境：

- 完成任務前自我審查。
- 開 PR 前檢查差異。
- 修正 Codex 或人工審查意見後再次確認。

檢查重點：

- 是否混入無關檔案。
- 是否有風險邏輯變更。
- 是否缺測試。
- 是否違反 MVP 範圍。
- 是否新增不必要相依套件或重構。

### GitHub Plugin

適用情境：

- 建立或檢視 GitHub Issue。
- 依 Issue 建立 feature branch。
- 檢查 PR 狀態與差異。
- 補充 PR description。
- 回覆 review comment。

使用規則：

- 不要直接 push 到 `main`。
- 每個 PR 盡量對應一個 Issue。
- PR description 應包含摘要、測試結果、手動測試步驟與剩餘風險。
- 修 review comment 時，只處理本次任務相關內容。
- 涉及遠端操作前，若需要使用者授權或登入，先清楚回報。

## 測試期待

回報完成前請執行相關測試。

後端或 API 變更應測試：

- 有效訂單建立。
- 無效請求 payload。
- 缺少必填欄位。
- 空購物車。
- 必填商品選項未選。
- 由伺服器計算價格。
- 訂單狀態轉換。

前端變更應測試：

- 手機版面。
- 菜單瀏覽。
- 分類切換。
- 商品詳情彈窗。
- 數量加減器。
- 必填選項驗證。
- 購物車流程。
- 結帳步驟 1 與步驟 2。
- 錯誤狀態。
- 載入狀態。
- 空狀態。

資料庫變更應測試：

- Migration。
- Seed data。
- 關聯完整性。
- 必填欄位。
- 預設值。

如果無法執行測試，請清楚說明原因。

## 安全與資料完整性規則

訂單是商業關鍵資料。

- 不要在未驗證的情況下信任前端提供的 LINE 使用者身分。
- 不要把 secrets 暴露到前端。
- 不要 commit `.env` 檔。
- 不要寫死 API keys。
- Admin API 不可在沒有 authentication 的情況下公開。
- 在伺服器端驗證所有訂單建立輸入。
- 不要只依賴前端驗證。
- 訂單編號必須唯一。
- 訂單總金額應由伺服器計算。
- 不要信任前端計算的價格。
- 訂單品項必須儲存商品名稱、商品選項與價格快照。
- 菜單價格變更不應影響歷史訂單。
- 菜單資料應為商家可配置資料，不得綁死參考截圖內容。
- 已取消訂單仍應保留在後台歷史紀錄中。

## LINE LIFF 規則

LIFF 相關實作應考慮：

- LIFF app 可能在 LINE 內建瀏覽器中執行。
- UI 必須在手機上好用。
- 客人身分應使用 LINE LIFF 流程。
- 敏感的使用者身分處理應使用伺服器端驗證。
- 不要假設 LIFF API 在 LIFF 環境外也可用。
- LIFF 初始化失敗時，提供 fallback 或清楚的錯誤訊息。

在 `docs/LINE_SETUP.md` 存在之前，或使用者明確要求建立之前，不要實作 LINE LIFF 設定。

## Codex 協作規則

Codex 作為獨立審查者與第二意見來源。

正常流程：

- Claude Code 實作功能。
- Claude Code 執行測試並自我審查。
- 建立 Pull Request。
- Codex 審查 PR。
- Claude Code 只修正 Codex 提出的 Must Fix 問題。
- 使用者手動測試 LINE LIFF 流程。
- 通過後 merge。

不要讓 Claude Code 和 Codex 同時修改同一個分支。

修正 Codex comments 時：

- 先修 Must Fix。
- 除非使用者要求，否則不要處理 Nice to Have。
- 不要重構無關程式碼。
- 說明每一個 Codex comment 如何被處理。

## Git 流程
